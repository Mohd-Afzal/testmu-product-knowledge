Part 1 · Software foundations

Front end, back end & UI/UX

Every app a customer ships has two halves and one feeling. Get these straight and the rest of the module clicks.

Front end

The part users see and touch — buttons, screens, visuals. Built with HTML, CSS & JavaScript. Think of it as the shop window.

Back end

The hidden engine — server, database & application logic — that makes things work. Think of it as the kitchen.

Restaurant analogy: the front end is the waiter taking your order; the back end is the chef preparing it. The front end requests data, the back end processes and returns it. A great app needs both working together seamlessly.
UI vs UX — the difference reps confuse most
UI (User Interface)UX (User Experience)
What you see and touch — buttons, colours, layout.How it feels to use — easy, enjoyable, frustrating.
Door analogy: the handle.Door analogy: how smoothly the door opens.
UI testing = does every button work and respond?UX testing = is the whole journey smooth and intuitive?

Why it matters for you: TestMu AI lives mostly on the front-end side of quality — making sure what users see and do works across every browser and device. We'll go deep on that in Part 3.2.

Part 1 · Software foundations

APIs — how systems talk

API = Application Programming Interface. It's the contract that lets two pieces of software talk without either knowing the other's internal mess.

Restaurant analogy: you (an app) want a service. The kitchen (another app) provides it. The waiter is the API — they carry your request and bring back a response, and you never need to see the kitchen.
How an API call works
  • A request has a server endpoint URL + a request method (usually HTTP).
  • A response has three parts: a status code, a header, and a body (the data, in JSON or XML).
  • You've seen status code 404 — "URL not found".

Real example: Apple's weather app doesn't build a global weather network — it calls weather.com's API. APIs let companies reuse data and protect their internal systems. That's also why API testing (Part 3.3) matters: if the contract breaks, the app silently breaks.

Part 2 · How software is built

The SDLC — six phases

SDLC = Software Development Life Cycle: the structured process for taking software from idea to live product. It relies on planning rather than "a roll of the dice." Conceived by Winston W. Royce in the 1970s.

1 · Planning & requirements

Define purpose, scope, needs. Output: the SRS (Software Requirements Specification).

2 · Design

The technical "blueprint" — architecture, UI, tech stack. Output: the DDS (Design Document Specification).

3 · Implementation

Developers write the actual code. Usually the longest phase.

4 · Testing

QA runs test cases; bugs go back to devs until it's stable. This is where the STLC lives →

5 · Deployment

Released to production for real users — increasingly automated via DevOps.

6 · Maintenance

Monitor the live app; fix bugs, improve, ship updates.

Construction analogy: Planning = "I want a 3-bedroom house." Design = architect's blueprints. Implementation = pouring concrete. Testing = inspectors checking wiring & plumbing. Deployment = handing over the keys. Maintenance = fixing a leaky roof a year later.

Part 2 · How software is built

Agile, Waterfall & DevOps

The same six SDLC phases get organised in very different ways. Which one a customer uses changes how — and how often — they test.

FeatureWaterfallAgile
ShapeOne long, linear sequenceMany short cycles called Sprints (1–4 weeks)
RequirementsFixed & fully defined up frontExpected to evolve with feedback
TestingOnly after building is completeContinuously, every sprint
RiskHigh — a late error means rolling back to stage oneLower — issues caught early via frequent testing
Best forFixed-requirement projects (gov, defence)Startups, consumer apps, fast-moving markets

The Agile Manifesto (2001) was a direct response to Waterfall's rigidity. Agile umbrella frameworks: Scrum (sprints), Kanban (visual boards), Lean (Build-Measure-Learn), XP (test-driven development, pair programming).

DevOps — the modern evolution

DevOps = Development + Operations merged into one collaborative team. It's visualised as an infinity loop (Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → back to Plan). It uses CI/CD to shrink the SDLC loop from weeks → minutes. Testing stops being a phase and becomes continuous.

CI/CD & environments

CI/CD is the "smart conveyor belt" that moves code from a laptop to production with automated checks at every step.

CI — Continuous Integration

Automatically check & merge code changes into a shared repo, running tests on every commit.

CD — Continuous Delivery

Validated code is prepared for release, awaiting human approval to ship.

Continuous Deployment

Goes one step further — ships to production with no human approval.

Part 3 · How software is tested

The Software Testing Life Cycle (STLC)

Here's the section that was missing. The STLC is the testing equivalent of the SDLC — it lives inside the SDLC's testing phase. Understand this and you understand the entire problem space TestMu AI sells into.

SDLC (the parent)STLC (the child)
Builds the product, start to finish.Verifies & validates the product within it.
Whole technical team.Mainly the QA team.
Phases: Planning → Analysis → Design → Development → Testing → Maintenance.Phases: Requirement Analysis → Test Planning → Test Case Dev → Environment Setup → Execution → Closure.
House analogy: SDLC is building the whole house. STLC is the home inspection inside that build — planning what to check, testing the wiring & water pressure, and reporting cracks before the owner moves in.
Verification vs validation (reviewers love this distinction)
Verification

"Are we building the system right?" — does it meet the technical spec?

Validation

"Are we building the right system?" — is it what the user actually needs?

The six STLC phases
1 · Requirement analysis

QA studies requirements to find what's testable. Output: queries list + automation-feasibility report.

2 · Test planning

The most important phase. Test Manager defines strategy, effort & cost. Output: the Test Plan.

3 · Test case development

Write detailed test cases + data. Use a traceability matrix for 100% coverage.

4 · Environment setup

Configure the test infrastructure; run smoke tests to confirm it's ready.

5 · Test execution

Run the tests; mark Pass/Fail/Blocked; log defects to developers; retest fixes.

6 · Cycle closure

Evaluate coverage & quality, hand over to maintenance, record lessons learned.

Part 3 · How software is tested

Frontend testing explained

This is where TestMu AI does most of its work, so it deserves its own section. Frontend testing = making sure everything the user sees and does actually works — on every browser, device and screen size.

Why it's uniquely hard: digital fragmentation

A modern web app must work across 3,000+ browser/OS combinations and 10,000+ real device permutations. Testing all of that by hand is impossible — this is the core problem the whole industry (and TestMu AI) exists to solve.

Cross-browser testing

Does the site render & behave correctly in Chrome, Safari, Edge, Firefox… and old versions of each?

Cross-device / responsive

Does it work on real iPhones, Androids, tablets and desktops at every resolution?

Visual / UI testing

Do buttons, layouts and pixels look right? Catch visual regressions a functional test would miss.

UX / accessibility testing

Is the journey smooth and usable by everyone, including people using assistive tech?

Types of testing

Tests are organised by level (how much of the app) and by approach. Know these names cold.

The four levels
Unit

The smallest piece (one function/module). Done during coding by developers.

Integration

Do combined modules work together correctly?

System

The whole integrated app tested against requirements. Includes Alpha (internal) & Beta (friendly customers) testing.

Regression

Did a new change break something that already worked? Run in the maintenance phase — and the #1 thing customers automate.

Part 3 · How software is tested

Testing jargon — the words on every call

These are the terms a QA prospect will use without explaining. Don't get caught off guard.

Test case

A set of steps to verify one feature — purpose, steps, expected result.

Bug / defect

An error causing wrong behaviour. Origins: coding mistakes, bad requirements, environment changes.

Test coverage

% of code/functionality exercised by tests. Low coverage = bugs slip through.

Flaky test

A test that passes/fails inconsistently without code changes — the "maintenance nightmare".

Regression suite

The battery of tests proving old features still work.

Parallel testing

Running many tests at once to slash execution time.

Release velocity

How fast updates reach users = (releases ÷ time). Slow velocity loses to competitors.

Traceability matrix

Links test cases to requirements so nothing goes untested.

Test data management

Creating/masking/subsetting data for realistic tests.
